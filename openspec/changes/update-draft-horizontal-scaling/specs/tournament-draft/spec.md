## ADDED Requirements

### Requirement: Draft Update Coalescing
The draft engine SHALL coalesce state broadcasts per session to reduce CPU and network utilization under bursty pick activity.

- The engine SHALL coalesce multiple internal state changes into at most one `draftUpdate` per 50–100 ms window per session.
- The engine SHALL still persist authoritative state; coalescing only affects outbound fanout timing.

#### Scenario: Multiple picks in quick succession
- WHEN 5 players make picks within 50 ms in the same session
- THEN the engine SHALL emit at most one `draftUpdate` to that session within the 50–100 ms window

### Requirement: Snapshot on Join
The server SHALL provide an up-to-date draft state snapshot immediately after a participant joins the session room.

- After handling `draft:session:join`, the server SHALL emit a `draftUpdate` (snapshot) to the joining socket.

#### Scenario: Join mid-draft
- WHEN a participant joins `draft:s` while the draft is in progress
- THEN the server SHALL emit the current draft state to the joining socket within 250 ms

### Requirement: Room-based Broadcasting Semantics
The server SHALL deliver draft updates via Socket.IO rooms and cross-instance pub/sub.

- The local instance SHALL emit to `draft:<sessionId>`.
- For cross-instance delivery, the engine or server SHALL publish to the Redis channel with `instanceId`; subscribers SHALL avoid echo by checking origin.

#### Scenario: Single delivery across instances
- WHEN a `draftUpdate` is generated by any instance for `sessionId = s`
- THEN all participants connected to `draft:s` SHALL receive the update exactly once
